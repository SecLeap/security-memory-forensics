# x64dbg / x32dbg 实战手册

> 适用范围：隔离 Win11 分析 VM 中，对良性测试程序、公开教学样本或明确授权的二进制工作副本进行用户态动态调试，并把地址、模块、线程和内存区域线索回填到全量 RAM 镜像分析。调试会改变进程的时间与控制流，不能替代在未调试状态下采集的内存证据。

[x64dbg](https://github.com/x64dbg/x64dbg) 是面向 Windows 用户态的开源二进制调试器。使用 `x32\x32dbg.exe` 调试 32 位目标，使用 `x64\x64dbg.exe` 调试 64 位目标；不确定时可使用 `x96dbg.exe` 选择架构。仅使用官方发行包和内置功能；不安装来源不明插件、脚本或“反反调试”扩展。

## 获取与安装

### 1. 下载、校验与便携部署

1. 从 [x64dbg 官方 Releases](https://github.com/x64dbg/x64dbg/releases) 下载发布包，保存原始压缩包至 `C:\Lab\Installers\x64dbg\`。
2. 解压到当前用户具有写权限的隔离 VM 目录，例如 `C:\Lab\Tools\x64dbg\`；官方说明建议使用此类可写目录。
3. 记录发布标签、下载 URL、文件名、SHA-256、签名状态（如适用）、许可证和 VM 快照。

```powershell
Get-FileHash 'C:\Lab\Installers\x64dbg\<发布包>' -Algorithm SHA256
Get-ChildItem 'C:\Lab\Tools\x64dbg\x32','C:\Lab\Tools\x64dbg\x64' -Filter '*dbg.exe' |
  Get-FileHash -Algorithm SHA256
Start-Process -FilePath 'C:\Lab\Tools\x64dbg\x96dbg.exe'
```

案例期间冻结版本和插件目录。调试器配置、数据库/注释与断点文件属于分析派生物，应保存在案例目录或单独的工具配置快照中，不混入样本原件。

### 2. 最小验证

1. 用已知良性的 32 位与 64 位测试程序副本分别启动 x32dbg/x64dbg，确认架构选择正确。
2. 在入口点暂停后查看 CPU、内存映射、模块和线程窗口，保存一张基线截图。
3. 退出测试进程并恢复 VM 快照（如实验要求），记录工具版本、目标架构和验证结果。

不要用未知样本验证安装；不得通过调试器修改、补丁、重打包或导出生产程序。

## 证据准备与调试边界

```text
C:\Lab\Cases\LAB-001\
├─ 00-原始文件\             # 只读原件
├─ 01-工作副本\             # 仅此目录中的文件可被加载/调试
├─ 02-静态分析\
├─ 03-x64dbg-记录\
│  ├─ 截图\
│  ├─ 地址与断点记录\
│  ├─ 文本日志\
│  └─ 导出工作副本\
└─ 04-内存镜像\
```

调试前记录原件与工作副本 SHA-256、案例授权、VM 快照名、隔离网络方式、x64dbg 版本、目标位数、预期观察问题和开始时间。不要将原件拖入调试器；只从 `01-工作副本` 加载。

本手册允许：查看寄存器、反汇编、调用栈、模块、线程、内存映射、字符串和受控运行时状态；设置少量与分析假设相关的临时断点；截图与记录。

本手册禁止：反检测/规避安全控制、修改指令或内存、改变 PE、注入 DLL/代码、利用漏洞、绕过许可/DRM、针对未授权目标附加调试器、把任何样本或调试输出上传到第三方服务。

## 基础使用流程

### 1. 加载、架构与入口点

1. 根据静态分析结果选择 x32dbg 或 x64dbg；若不确定，先用 `x96dbg.exe` 选择，随后在记录中写明实际调试器。
2. 在 GUI 中通过 `File > Open` 加载工作副本，或从 x64dbg 启动时选择该副本；确认标题栏路径、文件短哈希和位数。
3. 首次暂停时记录当前 RIP/EIP、模块名、模块基址、入口点 RVA、线程 ID、时间和断点原因。
4. 将运行时地址写成 `模块基址 + RVA = VA`，例如 `0x00007FF6... + 0x1234 = 0x00007FF6...`；不要只记录一个没有模块/时间的绝对地址。

常用控制键仅用于受控样本：`F9` 继续运行、`F7` 单步进入、`F8` 单步越过、`F2` 在当前指令设置/移除临时断点。每次继续运行前先截图或记录当前状态，避免丢失可复现的暂停位置。

### 2. 断点与控制流观察

断点的目标是回答一个明确取证问题，例如“静态识别的入口点是否对应当前加载模块”或“特定线程是否进入某个已知函数范围”。

1. 先从静态工具（PEStudio、DiE、Ghidra、FLOSS、capa）获得候选模块、函数或字符串位置，并记录其地址类型。
2. 在同一模块、同一映像基址下换算为运行时地址，设置一个最少化的临时断点。
3. 命中后记录：断点地址、模块、RVA/VA、线程 ID、寄存器摘要、调用栈摘要、相邻指令/字节和时间。
4. 仅在完成记录后以单步方式观察有限控制流；出现异常、退出或未预期行为时停止并记录，不尝试通过补丁或反检测手法强行继续。

调试命中证明的是“在该调试会话中，该线程到达该地址”。它不等同于未调试执行时必然出现相同行为，必须用未调试状态下的内存镜像、Procmon/Noriben、PCAP 或服务端日志交叉验证。

### 3. 模块、线程与内存映射

| 视图 | 记录的线索 | 后续复核 |
| --- | --- | --- |
| Modules | 模块路径、基址、大小、加载时序（如可见） | Volatility 模块/进程 VAD、磁盘文件哈希、PEStudio/DiE |
| Memory Map | 起止地址、权限、类型、映射文件/私有区域 | Volatility VAD、MemProcFS 内存映射、线程起始地址 |
| Threads / Call Stack | 线程 ID、起始点、当前 IP、栈帧摘要 | Volatility 线程对象、VAD/模块归属和采集时间 |
| CPU / Dump | 寄存器、反汇编、有限字节窗口 | HxD 原始字节、Ghidra 函数/RVA 与镜像中 VA |
| Symbols / Strings | 模块内的可读名称或字符串定位 | FLOSS、PEStudio、原始字节与网络/文件日志 |

不在 x64dbg 中编辑 Dump、修改内存保护或使用内存分配/注入功能。若发现私有可执行区域、模块外线程起点或内存映射差异，先记录地址与时间，随即采集全量 RAM 镜像。

### 4. 截图、日志与派生转储

优先保存截图和文本地址记录。截图文件名包含案例号、PID、模块、视图和 UTC 时间，例如 `LAB-001_PID-1234_sample-x64_modules_20260711T031500Z.png`。

若在**受控工作副本**中导出模块/内存范围以便静态复核：

1. 记录导出前的 PID、模块/VAD 范围、VA、长度、时间、调试器版本和导出功能/路径。
2. 将导出物保存到 `03-x64dbg-记录\导出工作副本\`，计算 SHA-256。
3. 明确标注该导出物是“调试会话派生物”，不能代替 WinPmem 全量镜像或由 Volatility/MemProcFS 产生的可验证导出物。
4. 用 HxD、PEStudio、DiE、FLOSS 或 capa 分析副本时，保留输入哈希和来源链。

不要使用 Scylla、Imports Fixer 或其他修复工具把调试导出物伪装成原始文件；如确有教学解包需求，必须在单独、明确授权的实验步骤中进行，并与取证证据分离。

## 与内存取证的联合分析

### 场景一：模块基址与镜像地址对齐

1. 调试会话中记录目标 PID、模块路径、基址、入口 RVA、当前 VA 和时间。
2. 立即按 [WinPmem 实战手册](14-WinPmem.md) 采集全量 RAM 镜像，并计算 SHA-256。
3. 按 [Volatility 3 实战手册](../02-Volatility3-实战手册.md) 检索相同 PID 的模块、VAD、线程和命令行；用 MemProcFS 交叉查看进程模块/内存映射。
4. 用 `运行时 VA - 模块基址 = RVA` 对齐 Ghidra/PEStudio 的静态位置。地址不同先检查 ASLR、重定位、模块版本、进程重启和采集时间差。

### 场景二：私有可执行区域与线程起点

1. x64dbg 中仅记录私有可执行区域范围、权限、所属线程和起始地址，不修改内容。
2. 采集镜像后，在 Volatility 3 中验证该范围的 VAD、保护属性、线程和周边模块关系。
3. 对可验证导出物用 HxD 复核 `MZ/PE` 头或裸字节边界；必要时用 DiE/FLOSS/capa 提供静态线索。
4. 结论区分“调试观察到的地址”与“镜像中由独立解析器确认的对象”。

### 场景三：静态字符串到受控网络会话

1. FLOSS/PEStudio 发现的域名、路径或协议字符串先作为假设，定位其所属模块/函数范围。
2. 在 x64dbg 中仅观察该函数范围是否被目标线程到达，并记录时间/地址。
3. 使用 Fiddler、Wireshark、FakeNet-NG 或 Debian 服务日志记录同一时间的受控通信。
4. 采集镜像并以 Volatility 3 的进程/socket/内存对象补足关联。没有独立网络证据时，不把字符串或断点命中写成通信事实。

## 常见问题与排错

| 现象 | 处理方式 |
| --- | --- |
| 选错 x32/x64 调试器 | 停止会话，记录目标架构与错误；从工作副本重新以正确架构加载 |
| 地址与 Ghidra/Volatility 不一致 | 区分 RVA、VA、文件偏移和模块基址；排查 ASLR、重定位、进程重启、镜像版本与采集时间 |
| 断点未命中 | 记录断点地址、模块、条件与运行时长；检查是否设置在正确模块/会话，不通过补丁强行触发 |
| 目标异常退出或行为改变 | 调试可能影响时序；停止、保留记录并在干净快照中重测，和未调试镜像结果分开报告 |
| 防病毒告警或插件提示 | 仅从官方源重新校验，禁用未知插件/脚本；不在生产系统设置永久例外 |
| 需要修复/解包导出物 | 这超出本取证手册；保留原始派生物，另建获授权的教学实验，不混淆证据来源 |

## 实战检查清单

- [ ] 已从官方发行包部署 x64dbg，并记录版本、SHA-256、插件状态与 VM 快照。
- [ ] 调试对象是隔离 VM 中的工作副本或授权测试进程，架构已确认。
- [ ] 已记录 PID、模块基址、RVA/VA、线程 ID、时间、断点原因和截图。
- [ ] 未使用反检测、补丁、内存写入、注入、进程控制或未知插件/脚本。
- [ ] 关键调试线索后已采集全量 RAM 镜像，且以 Volatility 3/MemProcFS 独立复核。
- [ ] 任意导出物均标注为调试会话派生物，并已建立哈希与来源链。
