# FTK Imager 实战手册：硬盘镜像、内存镜像与 VMDK 分析副本

> 适用范围：已获授权的 Windows 终端/实验虚拟机中，使用 Exterro FTK Imager 制作磁盘与内存镜像，并将**原始格式的磁盘工作副本**转换为 VMDK，供隔离虚拟化分析使用。原始证据、主镜像和 VMDK 分析副本必须分离；绝不从原件或主镜像启动虚拟机。

[Exterro FTK Imager](https://www.exterro.com/ftk-downloads)可创建硬盘、可移动介质和逻辑卷的取证镜像，并生成完整性哈希。本文所称“FTK-IMG”统一指 FTK Imager；不同年代的免费版、Imager Pro 与企业版的菜单、许可证和功能会不同，应以**本机版本的官方用户指南和随附帮助**为准。

## 范围、角色与不可逆风险

| 对象 | 角色 | 是否允许启动/写入 |
| --- | --- | --- |
| 现场/实验源盘 | 采集来源 | 仅通过合规只读流程读取；不在分析机挂载写入 |
| 主磁盘镜像（E01 或 Raw） | 原始取证主件 | 只读保存，不启动、不转换覆盖 |
| 原始 Raw 工作副本 | 格式转换输入 | 只读；只用于生成派生 VMDK |
| VMDK | 虚拟化分析派生物 | 默认不启动；若另获授权启动，必须独立隔离网络并使用不可变/快照磁盘 |
| 内存镜像 | 瞬时 RAM 证据 | 只读分析，不与磁盘镜像相互替代 |

活系统采集天然会改变系统状态。必须在授权、案例记录、最小干预和既定采集顺序下进行；无法保证不改变时，将采集工具版本、启动时间、操作人和影响说明写入采集记录。

## 获取与安装

1. 从 [Exterro FTK Downloads](https://www.exterro.com/ftk-downloads) 或组织已核验的软件库获取正确产品版本；原安装包存于 C:\Lab\Installers\FTK-Imager\。
2. 记录版本、产品分支、许可证状态、安装包/主程序 SHA-256、数字签名、下载 URL、安装时间、OS/补丁、时区和 VM/采集机标识。
3. 在独立、已快照的采集/分析机安装，用无害虚拟磁盘和无害内存测试验证界面、输出路径、哈希报告和只读挂载设置。不得以真实案件源盘作为“安装测试”。

    Get-FileHash 'C:\Lab\Installers\FTK-Imager\<安装包>' -Algorithm SHA256
    Get-AuthenticodeSignature 'C:\Program Files\Exterro\<FTK Imager 主程序>.exe' |
      Format-List Status,StatusMessage,SignerCertificate

## 证据目录与最小元数据

    D:\Cases\LAB-001\
    ├─ 00-采集记录\
    │  ├─ 磁盘采集记录.md
    │  └─ 内存采集记录.md
    ├─ 01-主证据\
    │  ├─ Disk-E01\
    │  └─ Memory\
    ├─ 02-工作副本\
    │  └─ Disk-Raw\
    ├─ 03-VMDK-分析副本\
    └─ 04-哈希与验证\

每次采集至少记录：案例号、授权范围、操作者/复核者、源设备或 VM 标识、磁盘型号/序列号/容量或内存容量、工具/版本/哈希、开始/结束 UTC、输出格式、分段大小、目标路径、使用的哈希算法、采集日志/错误、源与产物哈希、保管链与存储位置。

## 一、硬盘镜像制作

### 1. 采集前核对

1. 明确源类型：Physical Drive 用于整盘，Logical Drive 用于逻辑卷；错误选择会改变证据范围，先在记录中写明理由。
2. 确认源盘容量、序列号、连接方式和目标盘剩余空间；目标存储不得是源盘分区。
3. 对现场物理盘优先使用经验证的写保护和组织采集规范；对实验 VM，先记录虚拟磁盘配置、关机/快照状态和宿主路径。
4. 为案例填写 Evidence Item 信息，但不要把未经验证的案情描述写进哈希或文件名；所有时间同时记录本地时区与 UTC。

### 2. Create Disk Image 流程

1. 在 FTK Imager 中选择 Create Disk Image，选择已确认的源类型和源设备，再次核对容量/序列号/路径后继续。
2. 添加镜像目标。主证据优先选择组织认可的取证格式（常见为 E01）并启用该版本提供的完成后验证与哈希生成；如后续必须生成 VMDK，同时创建或导出一个**独立的 Raw（dd）工作副本**，不可把主 E01 直接当 VMDK 输入。
3. 设置案例目录、分段策略和输出命名，记录每一项选择。开始采集后不移动、重命名或写入源盘；完成后保存 FTK 的镜像日志和哈希报告。
4. 对每个分段文件、日志与主清单计算 SHA-256；镜像工具内置 MD5/SHA1/SHA256（视版本）验证与独立 SHA-256 记录均保留。

完成标准：源设备信息、镜像格式/分段、开始结束时间、FTK 验证结果、主件/日志 SHA-256 和保管链完整；出现读取错误、坏扇区、容量差异或验证失败时停止后续转换，保留原始错误信息。

### 3. 镜像后验证

1. 以 FTK 该版本的 Verify Image 功能或官方流程验证主镜像，保存验证时间与报告。
2. 使用另一套只读验证工具或 HashMyFiles/Get-FileHash 对镜像文件和报告进行独立哈希复核。
3. 原始镜像与工作副本使用不同目录、权限和命名；工作副本创建完成后同时记录源主件与副本 SHA-256。E01/Raw 格式不同的文件哈希不相同，应依据工具验证和虚拟扇区内容比对，而非期待文件哈希一致。

## 二、内存镜像制作

### 1. 采集顺序与边界

内存是易失证据。先记录采集前 UTC、本机时间/时区、登录用户、网络状态、目标主机/VM、可见的关键 PID 和当前磁盘状态；随后立即采集。不要为了让镜像“更干净”结束进程、清空日志、重启或安装额外工具。

FTK Imager 的内存采集入口/选项随版本而异，常见为 Capture Memory。只选择当前版本明确支持的 RAM 输出；若界面提供附带 pagefile 或其他磁盘对象的选项，必须在案例记录中逐项说明是否选择、原因及其与 RAM 镜像的关系。不要把 pagefile 或磁盘文件误标为全量内存镜像。

### 2. 受控采集步骤

1. 在 C:\ 或独立采集介质准备足够空间的目标目录；目标不能位于待采集源盘的同一风险范围内。
2. 启动当前已核验的 FTK Imager，进入 Capture Memory；确认主机标识、输出目录、文件名、附加选项和 UTC。
3. 立即执行采集，不进行其他无关操作；完成后保存工具日志/错误、输出文件大小和结束时间。
4. 对内存镜像和日志计算 SHA-256，复制出只读工作副本后再在 Volatility 3/MemProcFS 中分析。

    Get-FileHash 'D:\Cases\LAB-001\01-主证据\Memory\LAB-001-memory.raw' -Algorithm SHA256

内存镜像的工具版本、采集开始/结束时间、源主机状态和每次额外操作都属于关键解释信息。FTK Imager 采集成功不自动证明镜像完整、可解析或与磁盘镜像处于同一状态；分析时应记录操作系统、采集时点和 Volatility 识别/插件结果。

## 三、Raw 磁盘工作副本转换为 VMDK

### 1. 转换前原则

FTK Imager 的职责是取证采集与验证；VMDK 转换是独立的分析派生步骤。优先从同案的 Raw（dd）**工作副本**转换。若只有 E01，先根据当前 FTK 版本官方帮助将其以只读方式导出/生成受控 Raw 工作副本，并记录该导出操作；不要将 E01 文件扩展名改为 img 或让转换工具猜测格式。

转换用 [qemu-img](https://qemu-project.gitlab.io/qemu/tools/qemu-img.html) 必须对离线、未被虚拟机挂载或写入的文件执行。QEMU 官方警告：不要用 qemu-img 修改正在被虚拟机或其他进程使用的镜像。

### 2. 转换与内容验证

1. 将 Raw 工作副本置于 02-工作副本\Disk-Raw\，计算其 SHA-256、记录字节大小和扇区大小（若已知）。
2. 在隔离分析机安装经核验的 qemu-img；记录其版本、二进制 SHA-256 和完整命令。
3. 将输出写入 03-VMDK-分析副本\，保证输出卷有足够空间。以下以 raw 输入、VMDK 输出为例：

    qemu-img info -f raw 'D:\Cases\LAB-001\02-工作副本\Disk-Raw\disk.raw'
    qemu-img convert -p -f raw -O vmdk 'D:\Cases\LAB-001\02-工作副本\Disk-Raw\disk.raw' 'D:\Cases\LAB-001\03-VMDK-分析副本\LAB-001-disk.vmdk'
    qemu-img info 'D:\Cases\LAB-001\03-VMDK-分析副本\LAB-001-disk.vmdk'
    qemu-img compare -f raw -F vmdk 'D:\Cases\LAB-001\02-工作副本\Disk-Raw\disk.raw' 'D:\Cases\LAB-001\03-VMDK-分析副本\LAB-001-disk.vmdk'

4. 记录 compare 的返回状态、输入/输出路径、大小、格式、时间和 stderr/stdout。VMDK 文件哈希与 Raw 文件哈希必然不同；使用 compare 的虚拟扇区内容结果及各自 SHA-256 建立派生关系。
5. 生成 VMDK 后设置只读权限，并在隔离虚拟化平台中仅“挂载为附加数据盘”做初步检查。若后续需要启动，另建一次性 VM、禁用共享目录/剪贴板/USB/宿主网络、使用仅主机网络或完全断网，并在启动前创建不可变快照；此启动行为不属于主证据分析。

### 3. 转换失败与例外

| 现象 | 处理方式 |
| --- | --- |
| qemu-img 无法识别输入 | 核对是否真为 Raw、是否为完整镜像/分段、输入大小与来源；不得改扩展名试错 |
| compare 不一致 | 停止使用 VMDK，保留命令/输出；复核输入是否写入、格式、分段和工具版本，从新的只读工作副本重做 |
| VMDK 无法挂载/启动 | 记录虚拟化平台、控制器类型、启动模式与报错；不修复主件，不将失败解释为镜像损坏 |
| 源为加密盘 | 保留加密状态、元数据和授权范围；不要绕过或猜测密钥，转换不等于可读/可启动 |

## 四、磁盘—内存联合使用

1. 保留磁盘镜像与内存镜像各自的采集时间、工具、SHA-256 和保管链；二者是不同时间点、不同语义的证据。
2. 从内存中得到 PID、命令行、模块/VAD、网络对象和路径线索后，在磁盘主镜像/工作副本中只读查找相应对象；记录路径、文件哈希和时间差。
3. VMDK 仅可作为隔离复现/辅助查看副本。任何在启动后产生的文件、日志或内存状态都是新实验产物，绝不可回写为原始磁盘或内存事实。

- [ ] 已记录授权、设备/VM 标识、源范围、工具版本/签名/哈希、操作者和 UTC 时间。
- [ ] 磁盘主镜像、Raw 工作副本、VMDK 与内存镜像分别保存，均有 SHA-256 和保管链。
- [ ] 已保存 FTK 采集/验证日志、读取错误和独立哈希复核结果。
- [ ] 内存采集未附带无关操作，已记录采集时点、附加选项和 Volatility/MemProcFS 后续命令。
- [ ] VMDK 仅从 Raw 工作副本离线转换，已保存 qemu-img info/convert/compare 输出。
- [ ] 未从原件/主镜像启动 VM，未向源盘或主证据写入。
