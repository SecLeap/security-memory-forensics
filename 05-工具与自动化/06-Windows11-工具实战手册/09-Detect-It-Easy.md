# Detect It Easy（DiE / DIE-engine）实战手册

> 适用范围：隔离的 Windows 11 分析虚拟机、已获授权的样本副本，以及从内存镜像中恢复出的候选文件。DiE 用于**静态识别与线索分流**，不执行样本，也不能单独证明样本的真实行为。

项目当前以 [Detect-It-Easy](https://github.com/horsicq/Detect-It-Easy) 维护；用户常见的 `DIE-engine` 仓库地址仍可用于进入其发布页。官方提供图形版 `die`、命令行版 `diec` 和轻量图形版 `diel`；其识别结果结合签名与启发式规则，应作为待复核线索而非定论。

## 获取与安装

### 1. 获取与留存安装证据

1. 在干净、隔离的 Win11 分析机中，从 [官方 Releases](https://github.com/horsicq/DIE-engine/releases) 下载与系统架构匹配的 Windows 发布包；不要使用来历不明的整合包或截图中的旧版快捷方式。
2. 将原始压缩包放到 `C:\Lab\Installers\DIE\`，解压到 `C:\Lab\Tools\DIE\`。保留压缩包，不以下载后文件替换原件。
3. 在安装记录中写明下载 URL、发布日期/标签、文件名、架构、许可与 SHA-256；若发布页给出校验值或签名，须一并保存截图或文本。

```powershell
Get-FileHash 'C:\Lab\Installers\DIE\<发布包文件名>' -Algorithm SHA256
Get-ChildItem 'C:\Lab\Tools\DIE' -Recurse -File | Select-Object FullName,Length
```

4. 将工具目录纳入 Win11 基线快照。DiE 的规则数据库与程序版本会影响结果；升级程序或规则库前先导出旧版本号和旧输出，再创建新快照。

### 2. 验证可运行性

不同发布包的目录层级可能不同，以下示例以实际解压后含有 `die.exe`、`diec.exe` 的目录为准。

```powershell
Set-Location 'C:\Lab\Tools\DIE'
Get-ChildItem -File die*.exe
.\diec.exe 'C:\Lab\Samples\LAB-001\benign-test.exe'
```

完成标准：图形版可以打开**副本**，或 `diec` 能返回文件识别结果；记录 DiE 版本、规则库版本（如界面可见）与命令输出。若无可用的良性测试文件，不要用未知样本验证安装。

## 使用前的证据准备

1. 原始文件只读保存到 `C:\Lab\Samples\<案例编号>\00-原始文件\`；针对分析创建工作副本到 `01-工作副本\`。
2. 为原件和副本分别计算 SHA-256，记录来源、获取时间、文件大小、操作者与副本关系。
3. 将每次 DiE 输出保存到 `02-静态识别\`，文件名包含案例号、样本短哈希和工具版本，例如 `LAB-001_a1b2c3d4_DiE-3.x.txt`。
4. 不将受限样本上传到在线识别机器人、公共沙箱或第三方服务；静态识别在本地完成。

建议的最小记录字段：

| 字段 | 示例 | 用途 |
| --- | --- | --- |
| 样本 SHA-256 | `…` | 锁定被识别对象 |
| DiE 程序/规则版本 | `DiE 3.x / db 日期` | 使结果可复现 |
| 输入路径与来源 | `dumpfiles 输出 / PID 1234` | 建立证据链 |
| 格式与架构 | `PE32+ / x64` | 决定后续工具和视角 |
| 识别提示 | `编译器、壳、保护、熵异常` | 形成待验证假设 |
| 复核结果 | `PEStudio / HxD / 内存上下文` | 区分提示与事实 |

## 使用方法

### 图形界面：单文件初筛

1. 启动 `die.exe`，通过“打开”选择工作副本；不要双击样本，也不要从资源管理器的预览窗格打开它。
2. 先记录总览中的文件类型、位数、目标平台、入口点、编译器/运行时和签名提示。
3. 查看 PE、ELF 或 Mach-O 对应的结构信息；对于 Windows PE，重点记录节表、导入/导出、资源、TLS、证书和熵异常位置。
4. 记录规则命中的壳、保护器或编译器提示；它们可能因嵌入数据、改壳、截断或规则库差异出现误报、漏报或多个命中。
5. 需要定位时，将 DiE 提供的偏移、节名或入口点与 HxD/FileInsight 的原始字节复核；再用 PEStudio、CFF Explorer 或 Ghidra 验证结构性结论。

不应把“识别为某壳/某家族”直接写成结论。合格表述示例：`DiE 3.x 规则提示疑似 UPX；节布局与入口点仍待 PE 结构和内存转储复核。`

### 命令行：可留档的批量初筛

`diec` 适合对明确、有限的工作副本列表生成可重放的控制台记录。先在单个已知良性文件上确认路径与输出，再处理案例文件。

```powershell
Set-Location 'C:\Lab\Tools\DIE'
$case = 'LAB-001'
$sample = 'C:\Lab\Samples\LAB-001\01-工作副本\candidate.bin'
$out = "C:\Lab\Cases\$case\02-静态识别\$case-DiE.txt"
.\diec.exe $sample | Tee-Object -FilePath $out
Get-FileHash $sample -Algorithm SHA256 | Tee-Object -FilePath $out -Append
```

批量处理前应先列出输入清单与哈希，不要让通配符无意扫入原件、报告或整个磁盘。若当前版本需要其他参数或输出格式，以该版本发布包的帮助、`RUN.md` 和项目文档为准；保存实际执行命令与原始输出，不手工“整理”后替代原始结果。

### 结果解读与复核路径

| DiE 观察到的线索 | 下一步复核 | 可形成的结论边界 |
| --- | --- | --- |
| PE/ELF/Mach-O 类型、位数 | 文件头、架构与相应解析工具 | 可确认格式和架构，前提是头部完整 |
| 编译器/运行时提示 | 导入表、字符串、调试信息、Ghidra | 仅支持“与某工具链特征一致” |
| 壳/保护器提示、高熵节 | 节表、入口点、原始字节、解包后对比 | 支持“存在包装/压缩迹象”的假设 |
| `Unknown` 或格式不完整 | HxD、FileInsight、内存来源与边界 | 不能等同于恶意；可能是截断、裸数据或新格式 |
| 多条冲突命中 | 固定 DiE/规则版本，交叉用 PEStudio/CFF Explorer | 记录冲突，不挑选有利结果 |

## 与内存取证的联合使用

### 场景一：磁盘样本与运行时映像是否一致

目标是解释“磁盘副本看起来正常，但内存中的映像布局或模块信息不同”的原因，而不是依据 DiE 直接判断注入。

1. 在 Volatility 3 中记录相关进程的 PID、创建时间、命令行、模块、VAD 和可疑线程上下文。
2. 按 [Volatility 3 实战手册](../02-Volatility3-实战手册.md) 从可验证的文件对象或内存区域导出候选文件；给导出物计算哈希，并记录插件、参数、对象地址与导出时间。
3. 分别对磁盘工作副本和导出物运行 DiE，比较文件格式、架构、入口点、节布局及壳/熵提示。
4. 若差异存在，以 VAD 权限、映射方式、线程起始地址、原始字节和采集时间进一步解释；ASLR、重定位、截断或非完整 PE 都可能造成表象差异。

完成标准：报告明确列出两个对象的哈希和来源，并将“DiE 识别差异”与“已由内存上下文验证的事实”分开描述。

### 场景二：可疑 VAD 或进程转储中的候选代码

1. 先保留完整内存镜像与 Volatility 原始输出；不要以进程导出物代替完整镜像。
2. 对导出物使用 DiE 判断它更像 PE、ELF、Mach-O、脚本容器还是未识别裸数据，记录头部偏移与完整性问题。
3. 对被识别为 PE 的对象，以 PEStudio/HxD 检查 `MZ`、`PE\0\0`、节表、入口点和截断情况；对未识别对象保留原始字节、范围和所属 VAD。
4. 将结果回填至 PID、VAD 起止地址、内存保护属性、线程起始地址与网络时间线。只有这些相互支持时，才能描述为“该可执行格式对象与该进程内存区域相关”。

### 场景三：受控实验中的静态—动态—内存闭环

适用于仓库的 Win11/Ubuntu 隔离实验环境，材料限于良性测试程序、公开教学样本或明确授权材料。

1. 测试前：HashMyFiles 记录样本哈希；DiE、PEStudio 记录格式与静态线索。
2. 测试中：Noriben/Procmon 记录进程、文件和注册表事件；Wireshark 或 FakeNet-NG 记录受控网络证据。
3. 关键时间点：按采集手册制作全量内存镜像并计算哈希。
4. 测试后：Volatility 3 将 PID、命令行、VAD、模块、线程和 socket 与前两阶段时间线对齐；对恢复出的候选对象再次用 DiE 识别。

交付物至少包括：样本与镜像哈希、DiE 原始输出、PML/PCAP/模拟服务日志、Volatility 命令和完整输出，以及一张说明各证据如何相互印证的时间线。

## 常见问题与排错

| 现象 | 可能原因 | 处理方式 |
| --- | --- | --- |
| `diec.exe` 无法启动 | 解压不完整、架构/依赖不匹配、终端目录错误 | 重新校验官方发布包哈希；在工具目录中运行；记录系统版本与报错，不从非官方站点补 DLL |
| `Unknown` | 文件截断、裸内存页、格式不支持或规则未命中 | 检查文件大小、头部偏移、导出边界和原始字节；保留 `Unknown` 结论 |
| GUI 与 CLI 结果不同 | 程序/规则库版本不同、输入不是同一副本 | 比较 SHA-256、版本、数据库与命令；以同一版本复跑 |
| 壳或编译器误判 | 启发式/签名局限、嵌入内容或修改过的头部 | 用 PEStudio、HxD、CFF Explorer 和 Ghidra 交叉验证 |
| 防病毒告警 | 分析工具或发布包的行为特征触发检测 | 仅在隔离 VM 中，从官方来源重新核验；记录告警，不把未验证的例外加入日常主机 |
