# Regshot 实战手册

> 适用范围：隔离 Windows 11 实验虚拟机中，针对已获授权的无害测试程序或教学样本，建立注册表与指定目录的前后差异记录。Regshot 的差异是“两个观察时点之间发生了变化”的证据，不自动证明变化由目标进程造成。

[Regshot 官方项目页](https://sourceforge.net/projects/regshot/)提供该工具的发布入口。它适合做受控实验前后快照和离线差异报告；不用于回滚、删除或修复注册表，也不能替代原始事件记录、完整内存镜像或进程上下文。

## 获取与安装

1. 仅从官方项目页获取发布包；保留原始压缩包至 C:\Lab\Installers\Regshot\，记录下载 URL、发布版本、文件名、下载时间、数字签名信息（如有）和 SHA-256。
2. 解压到隔离 VM 的 C:\Lab\Tools\Regshot\，不要覆盖旧版目录或把程序加入生产主机 PATH。记录主程序哈希、系统版本、时区和 VM 快照标识。
3. 使用自建无害程序或手工创建的测试键值，验证能生成两份快照与 TXT/HTML 比较报告；不要用未知样本验证安装。

    Get-FileHash 'C:\Lab\Installers\Regshot\<发布包>' -Algorithm SHA256
    Get-ChildItem 'C:\Lab\Tools\Regshot' -Recurse -File | Get-FileHash -Algorithm SHA256

工具版本、操作系统版本、权限、快照范围和背景进程都会影响结果。一次案例内应固定版本和范围，实验前创建 VM 快照。

## 证据准备与范围

    C:\Lab\Cases\LAB-001\
    ├─ 00-实验基线\
    ├─ 01-Regshot\
    │  ├─ LAB-001-before.hiv
    │  ├─ LAB-001-after.hiv
    │  ├─ LAB-001-compare.txt
    │  └─ LAB-001-compare.html
    ├─ 02-Procmon\
    └─ 04-内存镜像与输出\

记录以下基线字段：案例号、VM 快照名、系统版本/补丁、UTC 与本地时区、当前用户、Regshot 版本/哈希、是否管理员、启用的注册表根键、附加目录范围、目标测试 PID、Procmon PML 名称与网络隔离状态。

### 范围原则

| 项目 | 建议 | 原因 |
| --- | --- | --- |
| 注册表范围 | 保持两次快照范围完全相同 | 否则差异无法比较 |
| 附加目录 | 仅选已定义的实验目录，例如 C:\Lab\TestOutput | 避免系统噪声与过量数据 |
| 时间 | 第一份快照后立即开始 Procmon，第二份快照前停止目标测试 | 便于与事件时间线对齐 |
| 权限 | 两次使用同一权限级别 | 权限不一致会造成可见性差异 |
| 背景活动 | 断网或维持固定隔离网络，关闭无关程序 | 降低更新、同步与防护扫描噪声 |

Regshot 不能锁定注册表；第二份快照前不要安装工具、更新软件、切换账户或重启系统，除非这些变化就是实验变量并已写入记录。

## 使用方法

### 1. 实验前快照

1. 启动 Regshot，确认案例号和工具版本；选择与案例一致的注册表/目录范围及输出格式。
2. 保存第一份快照为 LAB-001-before，放入案例目录。记录快照开始/结束时间和任何访问失败或权限提示。
3. 同时启动 Process Monitor，清空历史事件，设置只关注目标 PID/进程树及所关心的注册表路径；把原始 PML 写至 02-Procmon。
4. 对测试程序、启动命令和初始内存镜像（如采集）记录 SHA-256 与时间。

### 2. 实验后快照与比较

1. 停止目标程序或完成已授权测试，立即停止 Procmon 捕获并保存原始 PML。
2. 在 Regshot 中用**相同范围、相同权限**创建第二份快照，保存为 LAB-001-after。
3. 选择第一份和第二份快照生成比较报告；同时保存 TXT 与 HTML（如当前版本支持），不覆盖原始快照。
4. 在报告中保留新增、删除、修改的完整原始条目；另创建人工分析表，不能以手工摘要替代原始报告。
5. 对两个快照和报告计算 SHA-256，并记录生成命令/界面选项、完成时间和失败项。

建议人工分析表：

| 差异项 | Regshot 分类 | 前后值/路径 | Procmon 复核 | PID/时间 | 结论状态 |
| --- | --- | --- | --- | --- | --- |
| HKCU\...\Run | 新增 | 值名、数据 | RegSetValue 事件 | PID、UTC | 已交叉验证/仅差异 |

### 3. 解读边界

| Regshot 结果 | 可以说明 | 不能说明 | 必须复核 |
| --- | --- | --- | --- |
| 键/值新增 | 两次快照之间该对象可见且发生了新增 | 目标进程一定写入，或该项一定持久化 | PML 中的 RegCreateKey/RegSetValue、PID、时间 |
| 键/值修改 | 前后内容或元数据不同 | 修改内容一定被使用 | 写入进程、后续读取、相关文件/模块 |
| 键/值删除 | 前一快照可见、后一快照不可见 | 目标进程实施了删除 | PML、权限变化、用户/系统任务 |
| 没有差异 | 当前范围/时点内未观察到差异 | 程序未修改注册表 | 检查范围、权限、比较流程和 PML |

不要用 Regshot 的“比较”结果执行回滚或人工删除。报告中以“观察到差异”描述，只有同一 PID 的原始事件、时间和必要内存/文件证据支持时，才描述为“该进程写入”。

## 与内存取证的联合使用

### 场景一：启动项变化与进程上下文

1. Regshot 发现 Run/RunOnce、服务或其他配置项差异后，先保留完整差异条目和两个快照哈希。
2. 在 Procmon 中按同一路径、时间窗口和操作类型回查写入事件，记录 PID、进程树和命令行。
3. 在关键时点采集全量内存镜像；用 Volatility 3/MemProcFS 记录该 PID 的命令行、模块、VAD、句柄/文件对象（插件可用时）及相关注册表上下文。
4. 报告中分开写静态差异、Procmon 已观察写入，以及内存中关联进程对象；Regshot 不承担进程归因。

### 场景二：受控测试的前后闭环

测试前：快照一 + Procmon 启动 + 可选基线内存镜像。测试中：只运行授权材料，保存 PML。测试后：停止捕获 + 快照二 + 比较报告 + 内存镜像。交付物至少包括两个原始快照、原始比较报告、PML、输入/镜像哈希、PID 时间线和 Volatility 命令输出。

## 常见问题与排错

| 现象 | 处理方式 |
| --- | --- |
| 差异过多 | 检查是否范围过大、后台更新/防护扫描、两次权限或用户不同；记录噪声来源，不删除结果 |
| 报告中无预期变化 | 核对快照范围、第二次快照是否完成、PML 是否有对应写入；不能据此断言未发生 |
| 两份快照不可比较 | 使用相同版本、范围、权限和账户重新建立基线；保留失败快照与报错 |
| 访问被拒绝 | 记录路径、权限和错误；不要为获取结果而降低系统防护或修改证据对象 |
| 想用报告清理注册表 | 停止；本手册仅用于记录与分析，不授权回滚/删除操作 |
