# VS-b50cc9 样本分析步骤

## 二、静态分析步骤

### 步骤 1：使用 HashMyFiles 校验样本哈希

**操作：**

1. 打开 HashMyFiles，添加样本文件 20270612-003-b50cc9。
2. 查看并导出 MD5、SHA-1、SHA-256。
3. 将 SHA-256 与案例记录比对；一致后再进行后续分析。

**分析信息：**

~~~text
MD5:     81E1BE36D5AC581BE164BF89FE6943D0
SHA-1:   AC84F8656E4920D52BE45391A2633729836BF193
SHA-256: B50CC9B20CC7CC42AB77E1FB589C6C633D00D26AAFAE739577D7B50A80BA3586
~~~

**记录结果：** 样本哈希用于确认分析对象未被替换或损坏。

### 步骤 2：使用 PEStudio 查看 PE 基本信息

**操作：**

1. 使用 PEStudio 打开已校验的样本，不执行文件。
2. 查看 Overview、Version、Manifest、Strings、Sections 和 Indicators 页面。
3. 保存关键页面截图或导出报告。

**重点查看的信息：**

| 项目 | 本样本结果 |
| --- | --- |
| 文件类型 | PE32 Executable |
| 架构 | x86（32-bit） |
| Description | RegNow Download Manager |
| Product | GetRightToGo |
| 编译器 | Visual Studio 2005 |
| 熵值 | 6.412 |
| Overlay | 11,040 Bytes |

**字符串分析：**

在 Strings 或 Manifest 中搜索产品名、域名和下载相关字段，记录以下结果：

~~~text
GetRightToGo
GetRightToGo.com Downloader
RegNow Download Manager
www.GetRightToGo.com
www.digitalriver.com
livemetrics.digitalriver.com
~~~

**分析结果：** 文件版本资源和字符串指向 GetRightToGo / RegNow Download Manager，属于软件下载或下载管理组件的线索。

### 步骤 3：使用 Detect It Easy 识别编译器和壳

**操作：**

1. 将同一工作副本拖入 Detect It Easy。
2. 查看文件格式、位数、编译器和 Packer 信息。
3. 截图保存识别结果。

**重点查看的信息：**

~~~text
PE32
x86
Visual Studio 2005
~~~

**壳检查结果：**

未识别到以下常见壳或保护器：

~~~text
UPX
Themida
VMProtect
ASPack
PECompact
~~~

**分析结果：** 样本是未被上述工具识别为常见加壳的 32 位 PE 文件。未命中不代表绝对不存在自定义保护。

### 步骤 4：使用 capa 识别代码能力

**操作：**

1. 对已校验的样本运行 capa。
2. 保存完整输出、capa 版本和规则版本。
3. 从输出中提取 ATT&CK 技术、网络、注册表、文件和反分析相关能力。
4. 将规则命中标记为“能力线索”，不要直接写成实际行为。

**本样本识别的信息：**

| 类别 | capa 命中 |
| --- | --- |
| 系统与环境发现 | T1082 系统信息发现、T1083 文件发现、T1016 网络配置发现、T1614 地理位置发现 |
| 防御规避 | T1497 沙箱检测、T1027 混淆、T1112 注册表修改 |
| 收集 | T1056 键盘记录能力线索 |
| 网络 | DNS、Socket Send/Receive、HTTP Header |

**重点分析：**

- GetAsyncKeyState 和 SetWindowsHookEx：记录为键盘或 GUI 相关能力，需在动态分析中验证。
- IsDebuggerPresent、NtGlobalFlag、GetTickCount、鼠标活动检查：记录为反调试/反沙箱线索。
- DNS、Socket、HTTP：记录为网络通信能力，需结合 INetSim 日志判断实际用途。

## 三、动态分析准备步骤

### 步骤 5：准备隔离环境和监控工具

**操作：**

1. 恢复干净的 Windows 11 虚拟机快照。
2. 禁用桥接网络、NAT、共享目录、共享剪贴板和 USB 重定向。
3. 将虚拟机连接到 INetSim 控制的隔离网络。
4. 启动 INetSim，并记录启用的服务和日志目录。
5. 启动 Procmon，清空旧事件。
6. 启动 Noriben，确认其能调用 Procmon。
7. 在执行样本前记录基线：运行进程、启动项、相关文件夹和注册表项。

**Procmon 过滤条件：**

- Process Name is 样本文件名，Include；或
- PID is 样本 PID，Include。
- 重点保留文件系统和注册表事件。

## 四、动态分析步骤

### 步骤 6：使用 Procmon 分析文件行为

**操作：**

1. 开始 Procmon 捕获。
2. 执行一次样本，记录启动时间和 PID。
3. 在 Procmon 中筛选样本 PID。
4. 查看 CreateFile、WriteFile、SetDispositionInformationEx、DeleteFile 等事件。
5. 记录文件路径、操作、结果和时间顺序。
6. 停止捕获，保存 PML，并导出 CSV。

**本样本观察到的文件：**

~~~text
%APPDATA%\GetRightToGo\
  ├─ 20270612-003-b50cc9.data
  └─ 20270612-003-b50cc9.data0
~~~

**本样本的事件顺序：**

~~~text
Create → Write → Use → SetDispositionInformationEx/Delete
~~~

**分析结果：** 样本在 GetRightToGo 目录创建并写入 .data 文件，结束前删除文件；目录最终为空。这符合临时配置或下载缓存的行为。

### 步骤 7：使用 Noriben 分析注册表和行为摘要

**操作：**

1. 使用 Noriben 启动 Procmon 监控。
2. 执行一次样本并等待样本退出。
3. 停止 Noriben，查看生成的报告。
4. 在报告中搜索 Headlight、GetRightToGo、Run、RunOnce、Services 和 Task。
5. 将注册表创建、修改和删除事件与 Procmon 原始 PML 对照。

**本样本观察到的注册表项：**

~~~text
HKCU\Software\Headlight\GetRightToGo
  ├─ CustomizedApps
  └─ SharedConfig
~~~

**后续操作：**

~~~text
RegDeleteKey
~~~

**分析结果：** 样本创建 GetRightToGo 配置项，并在执行结束后删除；未观察到 Run/RunOnce、服务或计划任务持久化。

**注意：**

~~~text
HKLM\...\Services\bam\
~~~

中的记录是 Windows 的程序执行痕迹，应作为取证线索记录，不能判定为样本的恶意持久化行为。

### 步骤 8：使用 INetSim 分析网络行为

**操作：**

1. 在样本执行前启动 INetSim。
2. 记录 DNS、HTTP 和其他模拟服务日志。
3. 样本执行后按时间查看 INetSim session/report 日志。
4. 如已抓包，将 PCAP 与 INetSim 日志、Procmon 网络事件按时间进行比对。
5. 分别记录域名、请求路径、协议、响应和关联进程。

**本次观察到的主要 DNS 名称：**

~~~text
wpad.localdomain
dns.msftncsi.com
www.msftconnecttest.com
activity.windows.com
watson.events.data.microsoft.com
~~~

**分析方法：**

| 名称/类型 | 判断方法 |
| --- | --- |
| wpad.localdomain | Windows 自动代理发现（WPAD）背景流量 |
| dns.msftncsi.com、www.msftconnecttest.com | Windows 网络连接状态检测（NCSI） |
| activity.windows.com、watson.events.data.microsoft.com | Windows 遥测或错误报告相关流量 |

**本样本未观察到：**

~~~text
恶意域名
随机域名或 DGA 域名
IP 直连
HTTP/HTTPS 下载 Payload
IRC、SMTP 或 C2 会话
~~~

**分析结果：** capa 显示样本具有 DNS、Socket 和 HTTP 能力；但本次 INetSim 观察未发现恶意下载或远程控制通信。网络能力更符合软件下载器的功能。

## 五、结果汇总

| 工具 | 得到的关键信息 | 分析判断 |
| --- | --- | --- |
| HashMyFiles | 样本 MD5、SHA-1、SHA-256 | 确认分析对象。 |
| PEStudio | GetRightToGo、RegNow、Digital River 字符串和版本资源 | 指向下载管理组件。 |
| Detect It Easy | PE32、x86、Visual Studio 2005；未识别常见壳 | 明确文件结构和编译器线索。 |
| capa | 发现、网络、注册表、反分析和键盘相关能力 | 仅说明代码能力，需动态验证。 |
| Procmon | .data 文件被创建、写入、删除 | 临时文件或缓存行为。 |
| Noriben | GetRightToGo 注册表项创建后删除 | 临时配置，未发现持久化。 |
| INetSim | 以 Windows 网络探测和遥测为主 | 未发现恶意域名、载荷下载或 C2。 |

## 六、最终判断

通过 PEStudio、Detect It Easy、capa、Procmon、Noriben 和 INetSim 的分析，样本表现为 GetRightToGo / RegNow Download Manager 下载管理组件。

本次分析未发现恶意载荷释放、持久化、勒索、挖矿、木马行为或 C2 通信。capa 命中的键盘记录与反分析相关内容属于静态能力线索，动态分析未验证其实际执行。
